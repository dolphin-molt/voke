# Voke 真实设备与权限回归

> 回归日期：2026-07-22（Asia/Shanghai）
> 当前发布目标：`/Applications/Voke.app`，0.4.0 (9)
> 目的：验证真实外设、macOS 权限、重启恢复和 App 情景路由，不用单元测试代替真机结论。

## 本轮结论

当前已完成 USB 键盘、鼠标、现有授权、重启恢复和前台 App 路由回归。Pro Controller 在本轮执行时处于离线状态，因此手柄按键、摇杆和断连释放仍需在设备重新连接后执行。正式签名、公证和全新 Mac 首次安装仍属于发布链路，不在本地自签名回归中冒充完成。

回归中发现顶部状态只依据“映射总开关”显示绿色“映射运行中”，即使设备或权限不可用也会误报。本轮已改为五种真实状态：映射已暂停、等待设备连接、需要输入监控、需要辅助功能、映射运行中。

## 权限全流程边界

独立临时 Bundle ID 已用于检查未授权界面，不清空正式 Voke 的 TCC 记录。以下两步仍必须在新的、稳定签名身份或一台未安装过 Voke 的 Mac 上人工验收：

1. 从未授权状态打开“辅助功能”和“输入监控”，由用户在系统设置中手动启用。
2. 重启应用和覆盖升级后再次确认权限、映射、Profile 与设备学习结果。

不要通过清空正式 Voke 权限来模拟新用户；这会破坏当前开发机的真实使用环境，也不能替代 Developer ID 签名后的安装验证。

## 回归发现的产品风险

- HID 键盘和鼠标采用监听模式，原始按键或点击可能与映射动作同时发生。
- 离线手柄在重新启动后的设备导航中不会出现，用户缺少明确的“历史设备/等待重连”入口。
- 没有首次使用向导，权限、设备连接、映射总开关和动作测试目前分散在不同位置。
- 诊断能够说明权限和连接状态，但普通用户不会主动导出日志，关键故障应直接显示在主界面。

## 产品层后续优先级

### P0：让新用户第一次就能成功

1. 增加首次使用向导：选择设备、授予权限、套用一个基础模板、按键测试、完成。
2. 把顶部健康状态做成可点击的问题入口，直接解释“为什么不能工作”和下一步操作。
3. 导入含 Shell 命令的配置前展示命令清单并要求明确确认；默认不自动信任外部配置中的命令。
4. 正式版本采用 Developer ID 签名、公证和稳定升级身份，并在未使用过 Voke 的 Mac 上完成首装验收。

验收标准：不了解辅助功能、输入监控或 Profile 的用户，第一次打开后五分钟内能完成一个真实按键映射并看到目标 App 响应。

### P1：让它适合作为常驻工具

1. 增加菜单栏入口，提供当前设备、当前方案、暂停/恢复映射、打开主窗口和退出。
2. 增加可配置的入门模板，例如浏览网页、演示控制、AI 对话、剪辑辅助；模板只是用户可编辑的初始配置，不写死具体 App 逻辑。
3. 增加历史设备和重新绑定机制，解决同型号手柄顺序变化、HID 更换 USB 口后被识别成新设备的问题。
4. 导入配置前自动创建本地回滚点，导入失败或用户后悔时可以一键恢复。
5. 增加自动更新或至少清晰的版本检查和升级说明。

### P2：确认需求后再扩展

- 音频输入/输出设备展示和切换。
- 区域截图、窗口截图、录屏和剪贴板输出。
- MIDI、旋钮、Consumer Control 和 Stream Deck 类设备。
- 基础模式与高级模式，以及更真实的 3D/USdz 外观资源。

这些扩展都不应排在首装成功率、权限可理解性、配置安全和稳定升级之前。
